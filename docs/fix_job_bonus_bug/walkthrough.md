# ジョブボーナスが反映されないバグの修正

## 1. 発生したバグ・考慮漏れの事象と根本原因
### 事象
ユーザーから「攻撃力が 66 +32 となっており、戦士★9のボーナスが加算されていない」と報告があった。計算上はさらに+54のボーナスが加算されるべきであった。

### 根本原因
原因は以下の3つが複合して発生していた。
1. **ステータス計算ロジック（CharacterStatusService）でのN+1問題による履歴取得漏れ**
    `$character->jobHistories()->where(...)` を使用して現在のジョブレベルを取得しようとしていたが、正しく取得できず常にレベル1として計算されてしまっていた。
2. **Livewireコンポーネント（LeftSidebar）におけるモデルキャッシュの固着**
    LivewireのプロパティにEloquentモデル（`$character`）を持たせていたため、リクエスト間でシリアライズ・復元される際、DBを直接再取得せず古いリレーションデータ（ボーナス値が0だった頃のJobClass）がそのまま使い回されてしまっていた。
3. **デプロイ時の本番環境データベース更新漏れ（最重要）**
    ローカル環境のSQLiteでシーダーを実行してボーナス値を設定・確認後、ZIP転送方式（API経由）で本番サーバーへデプロイを行った。しかし、転送されたのはファイルのみであり、本番サーバーのMySQL上で `php artisan db:seed` 等のデータ更新コマンドが自動実行される仕組みになっていなかったため、本番環境のデータベースは古いまま（ボーナス0）となっていた。

## 2. 何をどう修正して直ったのか（具体的な解決策）
1. `CharacterStatusService` において、`$character->jobClass()->first()` のようにクエリビルダ経由で取得し直すことで、キャッシュを回避して常に最新のDB情報を読み込むように修正した。
2. `LeftSidebar.php`（Livewire）の `render` メソッド内で、`$baseCharacter->refresh()` を呼び出し、シリアライズされた古いモデルの状態を破棄して強制的に最新データをロードするようにした。
3. 本番サーバーのMySQLを更新するため、一時的に `routes/web.php` にArtisanコマンドを実行するエンドポイントを作成・デプロイし、外部からURLを叩くことで本番サーバー上で `JobSystemSeeder` を実行させ、その後エンドポイントを削除した。

## 3. 今後同じ問題を起こさないための教訓・注意点
- **本番環境とローカル環境のDBの乖離を意識する**: API経由等の手動スクリプトデプロイを採用している場合、マイグレーションやシーダーなどの「データベースの更新」が自動で走る保証はない。ローカルで直っても本番で直らない場合は、本番DBのデータが更新されているかを真っ先に疑うこと。
- **Livewireでのモデル保持に注意**: LivewireでEloquentモデルをパブリックプロパティに保持すると、リレーションを含めた古い状態がシリアライズされて固着する。ステータスなどのリアルタイム性が重要なデータを扱う場合は、`render` 内で都度 `refresh()` するか、DBから直接再取得するよう徹底する。
- **原因切り分けの重要性**: 今回、画面にデバッグ用の文字列（`Job:0 Lvl:9`等）を一時的に出力したことで、DBのレコードがおかしいのか、計算式がおかしいのか、レベルが取得できていないのかを明確に特定できた。問題が複雑な場合は、まず画面上で「どの変数がどうなっているか」を可視化することが早期解決の鍵となる。
