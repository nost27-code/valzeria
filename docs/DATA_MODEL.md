# DATA_MODEL.md

Purpose: compressed DB/data contract map.

| Entity/Table | Purpose | Key fields | Used by | Notes |
|---|---|---|---|---|
| users | <short> | <short> | <paths> | <short> |
| teams | <short> | <short> | <paths> | <short> |
| players | <short> | <short> | <paths> | <short> |
| contact_messages | Admin inquiry inbox entries from contact form and POP3 mailbox imports | `sender_email`, `subject`, `body`, `body_html`, `status`, `source`, `external_uid`, `received_at`, `attachment_path` | `ContactMessageManager`, `ContactMailboxImportService`, `AdminDashboard` | `body_html` is nullable and only populated for imported HTML email parts. |
| character_item_daily_supplies | Per-character daily supply depot ledger | `character_id`, `item_id`, `claimed_on`, `supplied_count`, `stocked_count` | `DailySupplyService`, `shop.supply` | `stocked_count` keeps recovery-item allowance that could not be carried due to the 10 item carry target; the UI shows carried stock, today's stocked amount, and today's unclaimed allowance as the depot breakdown. |
| character_exploration_states | Current exploration chain state | `character_id`, `area_id`, `exploration_point`, `chain_count`, `danger_rate`, `valmon_material_found`, `valmon_heal_used` | `ExplorationStateService`, `ExplorationService`, `ValmonService` | `valmon_heal_used` limits Lv75 Valmon emergency recovery to once per exploration chain. |
| characters | Player character state | `explore_stamina`, `explore_stamina_max`, `explore_stamina_updated_at`, `inn_rescue_streak`, `profile_card_background`, `profile_card_frame`, `profile_avatar_frame`, `profile_valmon_case`, existing battle/progress fields | `ExplorationStaminaService`, `ExplorationService`, `InnService`, `MainScreen`, `CityHeader`, `ProfileController` | 探索力制が有効な場合だけ通常探索/ボス挑戦で参照する。自然回復はアクセス時の遅延計算で、初期値は勝利数から算出した各自の最大探索力に揃える。探索力回復アイテムの使用分は `explore_stamina_max` を超えて `explore_stamina` に蓄積できる。`inn_rescue_streak` は宿代不足の救済宿泊連続回数で、通常料金宿泊時に0へ戻る。冒険者カードの背景/四角枠/キャラ枠/ヴァルモンケースは所持済みアセットから選択して保存する。 |
| character_adventurer_card_assets | Per-character owned adventurer card visual assets | `character_id`, `asset_type`, `asset_path`, `source`, `obtained_at` | `CharacterProfileService`, `ProfileController`, `CityHeader` | `asset_type` is `background`, `card_frame`, `avatar_frame`, or `valmon_case`; defaults are `images/profile/adventurer_card_bg01.webp`, `images/profile/adventurer_card_frame01.webp`, `images/profile/adventurer_avatar_frame01.webp`, and `images/profile/valmon_case01.webp`. |
| character_consumable_items | Per-character support item inventory | `character_id`, `item_key`, `quantity` | `AdventureSupportService`, `InventoryController`, 倉庫/補給商会 | 救助保険証、緊急救助要請、探索力の小瓶/薬など、補給商会の所持型支援アイテムを保持する。探索力回復商品は購入時にここへ入り、使用時に `characters.explore_stamina` を回復する。 |
| character_shop_limits | Per-character shop limits | `character_id`, `shop_item_key`, `limit_date`, `purchased_count`, `used_count` | `AdventureSupportService`, 補給商会 | 探索力回復商品などの日次購入制限を記録する。 |
| game_settings | Runtime operation/balance settings | `exploration.mode`, `exploration.stamina_*`, existing cooldown/drop settings | `GameSettingService`, `ExplorationStaminaService`, admin reward settings | `exploration.mode=cooldown` なら従来の連戦待機方式、`stamina` なら探索力消費方式。旧 `exploration.stamina_enabled` は判定に使わない。 |
| game_texts | Admin-editable UI/help text overrides | `key`, `value`, `description` | `GameTextService`, `FacilityTextManager`, `HelpTextManager`, `HelpContentService` | Help/案内所 keys use the `help.*` prefix; missing rows fall back to config defaults. |
| public_logs | Bottom chat/public log entries | `type`, `character_id`, `receiver_id`, `message`, `importance`, `created_at` | `PublicLogService`, `ChatLog`, `AdminChatManager`, `PublicLogManager` | `type=private` is used for personal chat/letters and is player-filtered by sender/receiver. Admins can search and delete selected rows from `/admin/public-logs`. |
| external_feed_items | External feed items already processed for player notifications | `source`, `guid_hash`, `guid`, `title`, `url`, `published_at`, `notified_at`, `raw` | `SyncNoteRssUpdates`, `CharacterNotificationService`/notification bell | Used to avoid duplicate official note RSS notifications. The sync command processes only the latest RSS item each run. |
| items | Equipment and item master | `type`, `max_enhance`, `affix_enabled`, stat bonus columns, rank metadata | `EquipmentEnhancementService`, `EquipmentAffixService`, shop/equipment views | Weapons, armor, and accessories with `max_enhance > 0` are currently raised to max +5 by enhancement balance migration. Enemy drop weapons/armors with `affix_enabled=true` can roll affixes when granted as drops. |
| job_classes | Job master definitions | `rank`, `max_job_level`, growth rates, master bonuses | `JobService`, `JobChange`, `JobManager` | `max_job_level` is normalized to 10 for all ranks; rank changes job EXP multiplier, not the master cap. |
| character_jobs | Per-character job progress | `character_id`, `job_class_id`, `job_level`, `job_exp`, `is_mastered`, `mastered_at` | `JobService`, `JobChange`, battle/exploration rewards | `job_level >= 10` is mastered under the current rules; migration clamps old progress above 10 to 10. |
| equipment_affix_prefixes / equipment_affix_suffixes | Drop equipment affix masters | prefix `affix_key`, stat/rate/weight; suffix `item_type`, `effect_type`, `species_key`, name, effect rate/weight | `EquipmentAffixService`, `DropService` | Seeder/migration-backed masters for shared prefix stat bonuses, weapon species killer suffixes, and armor species resist suffixes. |
| character_items | Per-character owned equipment instances | `affix_*_bonus`, `affix_quality`, `killer_species_key`, `killer_damage_rate`, `resist_species_key`, `species_damage_reduction_rate`, `affix_generated_at` | `CharacterItem`, `EquipmentAffixService`, `CharacterStatusService`, `BattleService` | Affix bonuses are stored on the dropped equipment instance and apply only while equipped. Weapon species killer and armor species reduction are PvE-only. |
| weapon_enhancement_recipes / armor_enhancement_recipes | Legacy enhancement material recipes | weapon `materials` JSON, armor `required_material_*`, enhancement level | old migrations; current `EquipmentEnhancementService` uses service recipes | Current +1..+5 enhancement recipes are defined in `EquipmentEnhancementService` so weapons, armor, and accessories share the same balance shape: fragments for +1/+2, stone+fragments for +3, high-purity stones for +4/+5, city materials for +4/+5, and `MAT_REFINING_CORE` for +5. |
| titles | Master title definitions | `name`, `category`, `unlock_type`, `target_type`, `target_id` | `TitleUnlockService`, `ValmonService` | `名相棒` uses `unlock_type=valmon_level`, `target_id=100` and is granted when a partner Valmon reaches Lv100. |
| npc_procurement_requests | Active/generated NPC material requests | `npc_id`, `requester_name`, `status`, `starts_at`, `expires_at`, template/batch fields | `NpcProcurementRequestGenerationService`, `NpcProcurementRequestService`, `NpcProcurementRequestController` | Requests are tied to tavern NPCs when generated; `requester_name` remains display text, while `npc_id` is the roster/tavern identity link. |
| npc_material_stocks | Materials held by NPCs from delivered procurement requests | `npc_id`, `material_id`, `quantity`, `last_received_at` | `NpcProcurementRequestService`, `NpcMarketListingService`, `MarketService` | Player deliveries add to NPC stock. Scheduled NPC market generation consumes this stock into NPC market listings; expired NPC listings return remaining quantity to this table. |
| market_listings / market_transactions | Player and NPC material market listings and sale history | `seller_type`, `seller_character_id`, `seller_npc_id`, material/price/quantity fields | `MarketService`, `MarketController`, `NpcMarketListingService` | `seller_type=character` keeps existing player market behavior. `seller_type=npc` uses `seller_npc_id`, has no player seller payout/notification, and is mixed into the normal buy list when the seller NPC is market-eligible. NPCs with `npc_rank=hero/legend` are excluded from NPC market selling. |
| arena_npc_rankings | NPC rank-battle entries mixed into arena standings | `npc_id`, `rank`, `level`, `battle_profile`, `wins`, `losses`, `is_active` | `ArenaNpcRankingService`, `ArenaNpcBattleService`, `ColosseumScreen`, `ColosseumRanking` | Uses existing `npc_master` portraits/names. Ranks share the same numeric space as `arena_rankings`, but are stored separately from player characters. Active NPCs are placed below the current player ranking band, with a minimum entry point of rank 51, so the player TOP10 and early growth lane stay protected. `level` is a display/battle-flavor level calibrated by NPC rank tier and increases by 1 on NPC wins up to Lv50, not a full player growth record; hero-tier NPCs are held with `is_active=false` and max around Lv50 until later release. `npc_rank=legend` and configured noncombat NPC IDs are excluded. |
| arena_npc_logs | Player-vs-NPC rank-battle result logs | `attacker_id`, `arena_npc_ranking_id`, `npc_id`, `is_attacker_win`, old/new ranks | `ArenaNpcBattleService`, `ColosseumScreen`, `BattleController` | Kept separate from `arena_logs` because `arena_logs.defender_id` is a required character FK. |
| arena_npc_auto_logs | Scheduled NPC rank-battle logs | `attacker_npc_ranking_id`, `attacker_npc_id`, `defender_type`, `defender_character_id`, `defender_npc_ranking_id`, `is_attacker_win`, old/new ranks | `ArenaNpcAutoBattleService`, `ColosseumScreen` | Records autonomous NPC challenges. Player-facing recent logs include rows where the player was the defender, shown as normal defense logs rather than a special NPC label. |

## Rules

- Schema changes require migration.
- Type changes require generated/manual type update.
- RLS/security behavior must be verified when auth-sensitive.
