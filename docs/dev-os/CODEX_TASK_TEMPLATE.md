# Codex実装指示書テンプレート

使い方: Full版をコピーして埋める。小修正（1〜2ファイル、DB変更なし、通貨に触れない）はLight版でよい。
「実装対象外」「DB変更」「完了条件」は空欄禁止。書くことがなければ「なし」と明記する。

---

## Full版（新機能・仕様変更・DB変更あり）

```markdown
# 実装指示書: <タイトル>

## 目的
<プレイヤー/運営にとって何が良くなるか。1〜3行>

## 背景
<なぜ今やるか。関連する既存仕様。参照: docs/DOMAIN_RULES.md の該当節、関連する過去の更新>

## 現状の問題
<今の挙動と、何が困るか>

## 実装対象
- <やること1>
- <やること2>

## 実装対象外（重要・必ず書く）
- <触ってはいけない周辺機能>
- <今回はやらない改善>
- 依頼範囲外のリファクタリング・文言変更は行わない

## 変更範囲
- 想定ファイル: <app/Services/..., resources/views/..., routes/web.php など>
- 想定しない範囲: <例: BattleService本体、輝石まわり>

## 既存仕様への影響
- <影響する機能と、壊してはいけない挙動。docs/dev-os/IMPACT_MAP.md の該当行を貼る>

## DB変更
- [ ] なし
- [ ] あり → migration内容: <カラム/テーブル/index>
  - $fillable / $casts 更新: <対象Model>
  - 既存データの扱い: <デフォルト値/バックフィル/既存プレイヤーへの補填>
  - ロールバック: down() で戻せるか / 戻せない場合の手順

## 画面/UI変更
- 対象画面: <URL/Blade/Livewireコンポーネント>
- スマホ幅（375px）での表示: <考慮点>
- 施設画面なら <x-layouts.facility> を使う

## バックエンド変更
- ロジックはService層へ（Blade/Controllerに計算を書かない）
- 状態更新後はPRGパターン（redirect()でGETへ）
- 同名エンティティは name 単独でなく area_id 等との複合条件で特定

## 管理者機能への影響
- [ ] なし
- [ ] あり → <管理画面に追加する表示/検索/ログ。誤操作防止の考慮>

## ログ・更新履歴への影響
- 公開ログ（下部チャット）: <流す/流さない。流すなら条件>
- battle_logs / gold_transactions 等の内部ログ: <記録内容>
- config/admin_update_summaries.php: <追記する/しない>
- docs同期: AI_CONTEXT / DOMAIN_RULES / FEATURE_STATUS のどれを更新するか

## エラーハンドリング
- <失敗時にユーザーへ何を見せるか。二重送信/連打時の挙動。トランザクション要否>

## セキュリティ注意点
- <本人以外のキャラを操作できないか。管理者専用データの露出。入力値検証>

## パフォーマンス注意点
- <N+1、全キャラ走査、キャッシュ要否。探索・戦闘は高頻度実行される前提>

## テスト観点
- <正常系/境界値/失敗系。php artisan test 可能なら対象テスト>

## 手動確認手順
1. <画面URLと操作>
2. <期待結果>
3. <DBで確認するテーブル・カラムと期待値>  ← 表示だけでなく必ずDBを見る

## ロールバック方針
- <migration:rollback可否 / 機能フラグ / 旧挙動に戻す最短手順>

## 完了条件
- [ ] <観測可能な条件1（例: ◯◯画面で◯◯が表示され、リロード後も保持される）>
- [ ] <条件2>
- [ ] php -l 通過、docs同期、admin_update_summaries 判断の報告

## 更新情報サマリ案
- category: added / changed / fixed / balance / internal
- title（15〜35字）: <案>
- detail（プレイヤー向け日本語）: <案>
```

## 指示書を書くときの原則

- **1指示書1目的**。「ついでに」を入れない。複数機能はタスクを分ける。
- 数値バランス（ドロップ率・価格・EXP・確率）は指示書に**具体値を明記**する。Codexに数値を決めさせない。
- 課金・輝石・ランキング・順位シフトに触る場合は、Full版必須＋QA_CHECKLIST.md の高リスク段階を指示書に添付する。
- 「〜っぽく」「いい感じに」は禁止。迷ったら現行画面のスクショか既存コードの参照先を貼る。
- 推測で書いた仕様には「仮置き」と明記し、Codexに確認を求めさせる。
