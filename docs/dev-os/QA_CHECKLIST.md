# 実装後QAチェックリスト

使い方: 段階1は全実装で必須。DB変更があれば段階2を追加。課金・通貨・ランキング系は段階3を追加。
AIに消化させてよいが、段階2・3の最終確認は人間が行う。
チェック結果は「✅/❌/該当なし」で報告し、❌はデプロイ前に必ず解消する。

---

## 段階1: 共通QA（すべての実装）

### 保存と反映
- [ ] 変更した値が**DBに保存されている**（表示だけの反映になっていない）。対象テーブルを実際にSELECTして確認した
- [ ] 画面リロード後・再ログイン後も状態が保持される
- [ ] Livewireの一時state（public property）だけに持たせて永続化を忘れていない

### 既存機能
- [ ] 通常探索→戦闘→勝利でEXP/職業EXP/ドロップが入る
- [ ] ボス挑戦が動く。通常探索にボスが混ざらない
- [ ] 変更画面の隣接機能（同じService/同じBladeを共有する画面）が壊れていない
- [ ] 同名の敵・素材を name 単独で引いていない（area_id等との複合条件か）

### UI
- [ ] PC幅で崩れない
- [ ] スマホ幅（375px）で崩れない。横スクロール・ボタン押下不能がない
- [ ] 施設画面は <x-layouts.facility>、「街へ戻る」は共通部品側にある
- [ ] 文言がプレイヤー向けとして自然（内部用語・英語カラム名が露出していない）
- [ ] MP表記でなくSP、職業レベルでなく職業ランク、ATK/DEF/MAG/SPR/SPD/LUK表記
- [ ] 奥義memo/必殺技descriptionを変更した場合、`php artisan valzeria:validate-job-arts` と validator 関連テストを実行した

### エラー・多重実行
- [ ] POST処理後にredirect（PRG）。ブラウザ更新で二重実行されない
- [ ] 連打・短時間連続実行がサーバー側で抑止される（探索系は既存ガードに乗せる）
- [ ] 失敗時（所持不足・条件未達・対象消失）にエラー画面でなく案内が出る
- [ ] 例外時にGold/素材/輝石が片側だけ減る中途半端な状態にならない（トランザクション）

### 履歴・docs
- [ ] config/admin_update_summaries.php の追記要否を判断し報告した
- [ ] docs/UPDATE_LOG.md への記載要否を判断した
- [ ] AI_CONTEXT / DOMAIN_RULES / FEATURE_STATUS の該当箇所を同期した（不要なら理由を報告）
- [ ] scratch_*.php 等の一時ファイルがcommit対象に混ざっていない

### 拡張性・β運用
- [ ] 数値がコード直書きでなくconfig/マスタ側にある（バランス調整で触る値なら必須）
- [ ] 後から弱体化・仕様変更したときに既存プレイヤーのデータが壊れない構造か
- [ ] 将来の拡張（新都市・新素材・新職）を塞ぐハードコードがない

## 段階2: DB変更がある場合の追加QA

- [ ] migrationが本番相当データで通る（既存行にNULL/デフォルト値問題がない）
- [ ] down() が書かれている。戻せない変更なら復旧手順が指示書に明記されている
- [ ] Model の $fillable / $casts を更新した
- [ ] migrationファイルを**リネームしていない**（リネームした場合、旧名のダミーファイルを作った）
- [ ] Seederで truncate する場合、外部キーチェック無効化→再有効化している
- [ ] Seeder実行後、代表レコードの値（件数・ID・drop_rate・価格等）を目視確認した
- [ ] 既存プレイヤーへの補填・初期値付与が必要か判断した（例: 新カラムの初期探索力、過去購入者への差分補填）
- [ ] indexが必要なクエリ（ランキング・ログ検索等）にindexを張った
- [ ] デプロイでmigrationが自動実行される前提で、**コードとスキーマの順序依存**がない（旧コード+新スキーマの瞬間に落ちないか）

## 段階3: 高リスク機能（課金・輝石・Gold・ランキング・PvP風・削除系）の追加QA

### 通貨・課金
- [ ] 有償輝石/無償輝石/合計の整合が保たれる。消費は無償優先などの既存ルールに従う
- [ ] Stripe Checkout / Webhook の二重処理防止（同一イベント再送で二重付与されない）
- [ ] 購入・消費が kiseki_transactions / gold_transactions にログされ、管理画面から追える
- [ ] 残高不足時に消費だけ・付与だけが起きない（ロック/トランザクション）
- [ ] 日次上限（輝石ドロップ・購入個数制限等）が日付境界で正しくリセットされる
- [ ] GoldからKisekiへの変換経路を作っていない

### ランキング・対戦
- [ ] 順位シフトのトランザクション整合（同時実行で順位重複・欠番が出ない）
- [ ] NPCランカーの保護ルール（TOP10プレイヤー枠等）を壊していない
- [ ] 敗者側への通知・公開ログが仕様どおり（流しすぎ・漏れすぎがない）
- [ ] キャッシュ（番付5分キャッシュ等）と実データの不整合が致命的でない

### 削除・送信系
- [ ] 削除前に確認ステップがある。誤操作で復元不能にならない
- [ ] 管理画面の一括操作に保護（管理人メッセージ保護のような仕組み）がある
- [ ] メール・通知の誤爆がない（対象者の絞り込み条件を確認した）

### 不正対策
- [ ] 新機能がGold/素材/輝石の無限増殖経路にならない（買値<売値、交換ループ、報酬の再取得）
- [ ] クライアントから渡るパラメータを信頼していない（エリアID以外の値を受けるなら検証）
- [ ] 異常値が出た場合に管理画面のログで検知できる

## デプロイ後スモークテスト（本番・人間が実施）

1. トップ表示→ログイン→街画面表示
2. 通常探索1回（EXP/ドロップ/探索力消費を確認）
3. 今回変更した画面の主要操作1往復
4. 管理画面ダッシュボード表示（エラーが出ていないか）
5. storage/logs/laravel.log に新規エラーがないか
