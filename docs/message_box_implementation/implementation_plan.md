# 手紙箱（メッセージ）機能の実装計画

ユーザーからの要望に基づき、現在「準備中」となっている手紙箱（受信箱・送信箱・新規作成）の機能を実装します。

## 概要と基本方針

1. **データソース**
   前回のチャット機能実装で `public_logs` テーブルに追加した `receiver_id`（個人チャット用）のログデータを「手紙」として活用します。
   - **受信箱**: `receiver_id` がログイン中のキャラクターIDと一致するログ
   - **送信箱**: `character_id` がログイン中のキャラクターIDで、かつ `receiver_id` が設定されているログ
   
2. **レイアウトの統一**
   プロジェクトナレッジのルール「施設レイアウトのUI・デザイン統一ルール」に従い、手紙箱は `<x-layouts.facility>` を用いた独立した専用画面（フルスクリーンレイアウト）として実装します。

3. **コンポーネント構成**
   `MessageBox` という一つの Livewire コンポーネントを作成し、内部で「受信箱 (inbox)」「送信箱 (outbox)」「新規作成 (create)」のタブ（状態）を切り替える設計とします。

---

## Proposed Changes (変更予定のファイル)

### [NEW] `app/Livewire/MessageBox.php`
- 手紙箱のロジックを管理する Livewire コンポーネント。
- プロパティ: `$currentTab` ('inbox', 'outbox', 'create' を切り替え)
- **inbox**: 宛先が自分（`receiver_id`）のログをページネーション付きで取得。
- **outbox**: 送信者が自分（`character_id`）で、かつ個人宛（`receiver_id != null`）のログを取得。
- **create**: 宛先のキャラクター一覧を取得し、メッセージを送信（内部的には `PublicLogService` のチャット送信処理と同じフローを呼び出すか、新規に保存処理を記述）する。

### [NEW] `resources/views/livewire/message-box.blade.php`
- 手紙箱のUIビュー。
- `<x-layouts.facility>` を継承し、背景画像とヘッダータイトル（「手紙箱」）を設定。
- タブナビゲーション（受信箱 / 送信箱 / 新規作成）。
- **受信箱・送信箱タブ**: 相手のキャラクターアイコン、名前、メッセージ本文、送信日時をリスト表示。受信箱には「返信する」ボタンを設置（新規作成タブに遷移し、宛先を自動セット）。
- **新規作成タブ**: 宛先（セレクトボックスで全冒険者から選択）、メッセージ本文（テキストエリア）、送信ボタン。

### [MODIFY] `routes/web.php`
- 手紙箱へのルートを追加。
  ```php
  Route::get('/message', \App\Livewire\MessageBox::class)->name('message.index');
  ```

### [MODIFY] `app/Livewire/MainScreen.php`
- `getLocationData` メソッド内の `message` 施設の定義を修正。
- `status` を `'active'` に変更し、ボタンの表示を「準備中」から「開く」に変更。
- 各ボタンのアクション（受信箱、送信箱、新規作成）から、`route('message.index', ['tab' => 'inbox'])` などのパラメータ付きで専用画面へ遷移できるようにルートを設定。

## User Review Required (ご確認事項)

> [!IMPORTANT]
> **「既読/未読」の管理について**
> 現在の `public_logs` テーブルには「既読フラグ」のカラムが存在しません。
> 厳密な未読件数の表示や、未読のみをハイライトするためには、新しく `is_read` カラム（boolean）をマイグレーションで追加する必要があります。
> **質問:** 今回の実装で「既読/未読」の管理（未読件数のバッジ表示など）までしっかり実装しますか？ それとも、まずはマイグレーション無しで履歴が読める・送れるという「簡易的な手紙箱」として実装してよろしいでしょうか？

> [!NOTE]
> **「保存件数・期間」について**
> 現在の仕様ではログの保存期間等は特に設定されておらず、古いログがそのまま残ります。
> 今回は「チャットの個人ログ＝手紙箱」という前提でデータを共通で一覧表示する設計となります。

上記の方針および確認事項について、問題がなければ実装（Execution）に進みます。特に「既読/未読フラグのカラムを追加するかどうか」をご判断ください。
