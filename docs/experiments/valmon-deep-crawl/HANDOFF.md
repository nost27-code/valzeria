作成日: 2026-07-03

## 位置づけ

この実装は `docs/experiments/valmon-deep-crawl/` 配下の静的プロトタイプです。
現行の `local_deploy.php` のフルデプロイZIPには含まれないため、デプロイ対象外です。

本番の Laravel route / Controller / Service / migration / Blade はまだ追加していません。

## 現在の主な仕様

### シーズン

- 金曜09:00開始、月曜09:00終了
- 週末カードシーズン
- シーズンが変わると `state.tower` を初期化

### 金貨

- 開始時に2枚解放
- 以後4時間ごとに2枚
- 月曜05:00の解放分で最大36枚
- 月曜09:00ちょうどに最後の金貨が出る問題を避けるため、最後の解放は終了4時間前
- 金貨は受け取り操作後に使用可能
- 金貨用途:
  - カード3択
  - TP +1

### TP

- 表示上は `TP（探検ポイント）`
- 旧称 `深層BP` / `調律ポイント` は画面文言から削除済み
- 地下へ降りた時に付与
- 同じ階層のTP報酬は1シーズン1回だけ
- 通常TP消費は `1, 2, 3...` と増加
- 装備枠拡張は別カウントで `1, 3, 5...`

### 能力UP

- `TP` アコーディオン内の各能力 `+` を押す
- 残りTPがある場合は即時割り振り
- 残りTPがない場合はモーダル表示
  - `攻撃を1上げますか`
  - `金貨1枚を消費`
  - `TPをN消費`
- 消費すると追加TPを1得て、その能力へ自動で1割り振る

### カード取得

- `カードを取得する` ボタンを押す
- モーダルで選択
  - `金貨1枚を消費して引く`
  - `TPをN消費して引く`
- 選択後はカード3択専用画面へ遷移
- 3択候補は生成時点で保存され、引き直し不可
- 1〜10階の初回クリアでもカード3択が発生

### デッキ

- 初期装備枠20枚
- 最大36枚
- デッキカードはヴァルモン表示枠内のアコーディオンに配置
- ソート/絞り込みあり
- 装備上限を超える場合はトースト通知

### 探索

- 探索力を1消費して進行
- 1階層あたりのイベント数は階層帯ごとに変化
- 探知が高いほど階段発見が早くなりやすい方向
- 宝箱発見にも探知が効く
- 階層突破後は `地下に降りる階段を見つけた！` をログに出す
- 次ボタンは `地下N階に降りる`
- 敗北時は1階に戻らず、同じ階層から再挑戦可能
- 敗北メッセージはヴァルモン視点で主人のもとへ戻る表現

## 2026-07-03 続き2: 新フロア到着時のログ整理

新しい階層のページに、前の画面ですでに知らせた/HUDに常時表示されている情報が
テキストとして重複表示され、そのフロア本来のイベント（罠/宝箱等）が埋もれる問題を修正。

- **カード獲得ログを削除**: `applyCardChoice()` から `addRunLog(...デッキ候補に加わった...)` を削除。
  カード取得はカード選択画面で確定した瞬間に `flash("「○○」を手に入れた！")` のトーストで
  すでに知らせているため、次のフロアのログに再掲しない。
- **階層到達ログを削除**: `awardDescentBp()` 内の `addRunLog("地下N階へ降りた。TP +N")` を削除し、
  `flash(...)` のトースト表示に変更。TP数はヘッダーの `TP` カウンターに常時表示されているため、
  本文ログとしては冗長だった。関数シグネチャも `awardDescentBp(run, floorNumber)` →
  `awardDescentBp(floorNumber)` に簡略化（run未使用のため）。既踏破フロアの再訪時（TP無し）は
  何も表示しない（早期return）。
- **温存した項目**: 一時TP選択の結果テキスト（例:「耳を澄ませ、探知の感覚を強めた。」）は
  そのフロアに滞在中ずっと表示される場所が他になく、プレイヤーの選択結果を示す一次情報のため
  ログに残した。「階段を見つけた！」note も文脈として残した（前ページでの主イベントの結果を
  引き継いで表示する設計のまま）。

修正後、新フロアのログ先頭は「(前フロアの)階段発見ログ」+「一時TP選択の結果」のみになり、
そのフロア自身のイベント（画像→詳細ログ）が主役として続く構成になった。

## 2026-07-03 続き3: 階段発見ログを最下部に移動

「地下に降りる階段を見つけた！」は、前フロアをクリアした"結果"として得られるものなので、
そのページのログの最下部（末尾）にあるべき、という指摘を反映。

- 従来: 前フロアのクリアで発生したこの note は、次フロアのページを開いた時に
  「leading notes（本文より前）」として画像・本文の上に表示されていた
  → 次フロアの内容より前に出ると、この階の出来事のように見えて紛らわしかった
- 修正: `leadingLogRange()` で対象範囲の log イベントを取得したうえで、
  `renderLeadingChoiceLogEvents()`（先頭表示、階段ログは除外）と
  `renderLeadingStairsLogEvents()`（新設、階段ログのみ）に分離。
  `renderRun()` の描画順を
  「画像 → 先頭ノート(階段以外) → 本文(戦闘/イベント枠) → 末尾ノート → 持ち越し階段ログ」
  に変更し、前フロアからの階段発見ログが常にページ最下部に来るようにした。
- そのフロア自身をクリアしたことで新規に発生する階段ログ（`renderTrailingLogEvents` 経由）は
  元々ページ末尾に出ていたため変更なし。両ケースとも最終的に最下部に統一表示される。

## 2026-07-03 続き4: 分かれ道選択中の階段ログ抑制 + 現在階層の明示

「分かれ道」等の選択待ちパネルが出ている画面で「地下に降りる階段を見つけた！」が
同時表示されると、"もう降りたのにこれから道を選ぶ"という矛盾した見え方になっていた。
また、選択パネルまでスクロールすると上部の階層プログレスバーが見えなくなり、
今どの階にいるか分かりにくかった。

- **階段ログの抑制**: `renderTrailingLogEvents(run, pageEvent, suppressStairs)` に
  `suppressStairs` 引数を追加。`renderRun()` で `run.pendingChoices.length > 0` の間は
  `suppressStairs=true` で呼び出し、同時に `renderLeadingStairsLogEvents` の呼び出しも
  スキップする。選択が全て解決されて新しいページに進むと、
  そのページの末尾に改めて表示される（情報は消えず、タイミングだけ遅延する）。
- **現在階層の明示**: `index.html` の `#choicePanel` 見出しに `#choiceFloorLabel` チップを追加。
  `renderChoices()` で選択パネル表示時に `現在地下${run.currentFloor}階` を設定。
  選択肢までスクロールしても、パネル自身に現在階層が常に見える。

## 2026-07-03 続き5: スコアを画面下部の「結果」表示に変更

スコアが `.battle-run-meta`（イベントタイプバッジと並び、パネル最上部）に表示されており、
本文を読む前に結果の数字だけ見えてしまっていた。「戦闘結果・イベント結果は下部にあるべき」
という指摘を反映し、階段ログと同様に結果系の情報は最下部へ統一。

- `index.html`: `#runScoreText` を `.battle-run-meta`（上部、イベントタイプバッジのみ残す）から
  `#runLog` の直後に新設した `.battle-score-result`（スコア ラベル + 値）へ移動。
  要素IDは変更していないため、`app.js` 側の `$("runScoreText")` 更新ロジックは無修正。
- `styles.css`: `.battle-run-meta` を単一列グリッドに簡素化。`.battle-score-result` を新設
  （上端に破線区切り、ラベル+大きめの緑数字、ログの最後の「結果行」として視認できるデザイン）。

表示順は最終的に: 画像/アリーナ → 詳細ログ → (階段ログ) → スコア、という「結果は末尾」で統一。

## 2026-07-03 続き6: VS対戦カードの中央寄せ崩れを修正

戦闘アリーナ（プレイヤー vs 敵のVS表示）で、リーフィン側の画像・名前が左端に張り付き、
左右非対称に見える不具合を修正。

- 原因: `.battle-scene-panel .combatant-card` に `display:grid` と `justify-items:center` を
  指定していたが、`grid-template-columns` を明示していなかったため、暗黙の1列トラックが
  コンテンツの内容量（shrink-to-fit）で幅決定され、カード全体の145px幅いっぱいに広がって
  いなかった。結果、`justify-items:center` が「狭い暗黙列の中」でしか効かず、見た目には
  左寄せのように見えていた。
- 修正: `.battle-scene-panel .combatant-card` に `grid-template-columns: 100%;` を追加し、
  暗黙列をカード幅いっぱいに広げることで、画像・名前・HPバーが正しくカード中央に来るようにした。
- `preview_eval` で `getBoundingClientRect()` を比較し、プレイヤー側・敵側それぞれの
  画像/敵ビジュアルの中心がカード中心と一致することを確認済み（360px/375px幅で確認）。

## 2026-07-03 続き7: TPでの「HP全回復」を削除

このゲームは「敗北しても同じ階層から再挑戦でき、敗北時にHPは自動で全回復する」設計
（`finishRun()` で `reason === "defeated"` の場合 `state.tower.currentHp = run.maxHp` を代入）。
そのため、TPを消費してわざわざHPを全回復するアクションは常に無料の自然回復（死に戻り）と
機能重複しており、実質的に選ぶ理由がない不要な項目だった。

- `app.js`: `bpActionMeta()` の `heal` エントリ、`spendBp()` の `heal` 分岐、
  `renderBpActions()` の `{ key: "heal", ... }` を削除（カード分類の `"heal"`＝回復カードは
  別機能なので変更なし）。
- `index.html`: ルール文中の「HP回復」への言及を削除・置き換え（TP用途の列挙、
  通常TP消費の例、攻略の考え方の一文）。
- `styles.css`: `.bp-action-grid` の列数を2列固定→1列（`minmax(0,1fr)`）に変更。
  残る項目が「装備枠拡張」1つだけになったため。
