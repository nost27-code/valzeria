# 冒険都市ヴァルゼリア フェーズ1 レアドロップ 実装指示書

## ファイル名

valzeria_phase1_enemy_drop_spec.md

---

## 1. 目的

本指示書は、冒険都市ヴァルゼリアに「敵撃破時の装備ドロップ機能」を実装するためのものである。

現在、迷宮探索、戦闘、EXP獲得、GOLD獲得、装備購入、装備変更は実装済みである。

次に、通常戦闘やボス戦で敵を倒した際、一定確率で装備を入手できるようにする。

この実装により、戦闘の目的を以下のように広げる。

```text
戦う
↓
EXPを得る
↓
GOLDを得る
↓
低確率で装備が落ちる
↓
拾った装備を装備する
↓
さらに強くなる
↓
もう一度探索したくなる
```

フェーズ1では、素材、鑑定、装備強化、マーケット取引は扱わない。
敵から直接、装備アイテムが落ちる仕様とする。

## 2. 実装対象

今回実装するものは以下。

* enemy_drops テーブル
* 敵ごとのドロップ設定
* ドロップ率設定
* 戦闘勝利時のドロップ抽選
* ドロップ装備の character_items 追加
* 戦闘結果画面へのドロップ表示

* 所持済み装備の扱い
* ボス専用ドロップ
* ドロップ用装備データ追加

今回実装しないものは以下。

* 素材アイテム
* 消費アイテム
* 装備強化素材
* 鑑定前アイテム

* 装備の個体値
* 装備売却
* 倉庫
* プレイヤー間取引
* オークション
* ガチャ

## 3. 基本方針

## 3.1 ドロップ対象は装備のみ

フェーズ1では、敵が落とすものは装備のみとする。

対象は既存の `items` テーブルに登録されている装備である。

ドロップした装備は、`character_items` に追加する。

## 3.2 店売り装備とドロップ装備を分ける

店売り装備は、GOLDを使えば確実に手に入る装備。
ドロップ装備は、敵を倒したときに低確率で手に入る装備。

この2つの役割を分ける。

店売り装備は安定成長用。
ドロップ装備は探索の楽しみ用。

## 3.3 強すぎる装備は出さない

フェーズ1のドロップ装備は、店売り装備より少し強い程度にする。

強すぎる装備を序盤で出すと、エリア進行やボス戦バランスが壊れるためである。

目安は以下。

```text
同レベル帯の店売り装備より、補正値が1〜3程度高い
```

## 4. DB追加仕様

## 4.1 enemy_drops テーブル

敵ごとのドロップ設定を管理する。

テーブル名。

```text
enemy_drops
```

カラム案。

| カラム                 | 型            | 内容             |
| ------------------- | ------------ | -------------- |
| id                  | bigint       | 主キー            |
| enemy_id            | bigint       | 敵ID            |
| item_id             | bigint       | ドロップする装備ID     |
| drop_rate           | decimal      | ドロップ率          |
| min_character_level | int          | ドロップ可能な最低キャラLv |
| max_character_level | int nullable | ドロップ可能な最高キャラLv |
| is_active           | boolean      | 有効か            |
| created_at          | timestamp    | 作成日時           |
| updated_at          | timestamp    | 更新日時           |

## 4.2 drop_rate の扱い

`drop_rate` はパーセントで扱う。

例。

| drop_rate | 意味   |
| --------: | ---- |
|      10.0 | 10%  |
|       3.0 | 3%   |
|       1.0 | 1%   |
|       0.5 | 0.5% |
|       0.1 | 0.1% |

内部実装では、0〜100の乱数で判定してもよい。

## 4.3 min_character_level

低レベルのうちに強い装備が落ちすぎないよう、最低キャラクターレベルを設定できるようにする。

フェーズ1では基本的に1でよい。

ただし、ボス装備や洞窟装備はレベル条件を付けてもよい。

## 4.4 max_character_level

フェーズ1では未使用でもよい。

将来的に、低レベル帯専用ドロップなどを制御するために用意する。

不要なら nullable でよい。

## 6. character_items への登録

敵から装備がドロップした場合、`character_items` にレコードを追加する。

登録内容。

| カラム           | 値            |
| ------------- | ------------ |
| character_id  | 戦闘したキャラクターID |
| item_id       | ドロップした装備ID   |
| is_equipped   | false        |
| equipped_slot | null         |
| acquired_from | drop         |

ドロップしただけでは自動装備しない。

戦闘結果画面に「装備する」導線を出すのは可。

## 7. 所持済み装備の扱い

## 7.1 方針

フェーズ1では、同じドロップ装備を複数所持できる仕様でよい。

理由は、将来的な売却、倉庫、強化素材化に拡張しやすいため。

ただし、現時点で売却がない場合、同じ装備が増えるだけになる。

そのため、今回の推奨仕様は以下。

```text
店売り装備：同一装備は1つまで
ドロップ装備：同一装備を複数所持可能
```

## 7.2 実装ルール

`items.is_shop_item = false` のドロップ装備は複数所持可能とする。

同じ装備が落ちても、毎回 `character_items` に追加する。

## 7.3 将来対応

将来的に売却を実装した場合、重複装備はGOLD化できるようにする。

## 8. ドロップ抽選仕様

## 8.1 抽選タイミング

ドロップ抽選は、戦闘に勝利した場合のみ行う。

敗北時は抽選しない。

```text
戦闘勝利
↓
EXP付与
↓
GOLD付与
↓
ドロップ抽選
↓
戦闘ログ保存
↓
結果表示
```

EXP、GOLD付与とドロップ抽選の順番は厳密でなくてよいが、勝利時のみ実行する。

## 8.2 抽選対象

抽選対象は以下。

```text
enemy_id = 撃破した敵ID
is_active = true
character.level >= min_character_level
max_character_level が null または character.level <= max_character_level
```

## 8.3 複数ドロップ設定がある場合

1体の敵に複数のドロップ設定を持たせることを許可する。

フェーズ1では、1戦闘で最大1個までドロップとする。

複数候補がある場合の推奨処理。

```text
1. 該当する enemy_drops を取得
2. sort_order または rarity 順で並べる
3. それぞれ drop_rate 判定
4. 最初に当選した1件を採用
5. それ以降は判定しない
```

ただし、この方式だと並び順の影響が出る。

より自然にするなら、以下。

```text
1. 各ドロップ候補ごとに当選判定
2. 当選した候補を配列に入れる
3. 当選候補が複数あれば、その中からランダムで1つ採用
```

推奨は後者。

## 8.4 1戦闘あたりのドロップ上限

フェーズ1では1個まで。

将来的に複数ドロップを実装する場合は、エリアやイベントで制御する。

## 9. DropService 仕様

## 9.1 役割

ドロップ抽選処理は `DropService` に集約する。

Controller や BattleService に直接書かない。

## 9.2 DropService の責務

* 敵に紐づくドロップ候補取得
* レベル条件チェック
* ドロップ率判定
* 当選アイテム決定
* character_items への登録
* 結果返却

## 9.3 メソッド例

```text
rollDrop(character, enemy)
grantDropItem(character, item)
```

## 9.4 返却データ例

ドロップなし。

```text
null
```

ドロップあり。

```text
{
  "item_id": 12,
  "item_name": "古びた短剣",
  "rarity": "rare",
  "character_item_id": 1001
}
```

## 9.5 トランザクション

ドロップ装備の付与は、戦闘結果保存と同じトランザクション内で行うのが望ましい。

最低限、以下が不整合にならないようにする。

* 戦闘勝利したのに装備付与だけ失敗
* 装備付与されたのに戦闘ログが残らない

可能なら、探索処理全体をトランザクションで囲む。

## 10. BattleService / ExplorationService 連携

## 10.1 実装位置

ドロップ抽選は、戦闘勝利後に行う。

推奨位置。

```text
ExplorationService
  ↓
BattleServiceで戦闘
  ↓
勝利判定
  ↓
EXP/GOLD付与
  ↓
DropServiceで抽選
  ↓
BattleLog保存
  ↓
PublicLog必要なら作成
```

## 10.2 BattleServiceには入れすぎない

BattleServiceは戦闘計算に集中させる。

ドロップ抽選はBattleServiceではなく、ExplorationServiceまたはBattleResult処理側からDropServiceを呼ぶ。

理由は、BattleServiceが肥大化しすぎるのを防ぐため。

## 11. battle_logs 追加仕様

## 11.1 dropped_item_id

個人戦闘ログにドロップ装備を記録する。

既存の `battle_logs` に以下を追加する。

| カラム                       | 型               | 内容                       |
| ------------------------- | --------------- | ------------------------ |
| dropped_item_id           | bigint nullable | ドロップした item_id           |
| dropped_character_item_id | bigint nullable | 付与された character_items.id |

`dropped_item_id` だけでもよいが、所持装備との紐づけを後から確認しやすいため、`dropped_character_item_id` もあるとよい。

## 11.2 ドロップなしの場合

どちらも null。

## 12. 戦闘結果画面への表示

## 12.1 ドロップなし

ドロップがなかった場合は、特に表示しなくてもよい。

または控えめに表示する。

```text
ドロップ：なし
```

フェーズ1では表示しなくてもよい。

## 12.2 ドロップあり

戦闘結果画面に、ドロップ装備を表示する。

表示例。

```text
アイテム獲得！

古びた短剣 を手に入れた！
```

レアリティも表示する。

```text
アイテム獲得！

[Rare] 古びた短剣 を手に入れた！
```

## 12.3 導線

ドロップ装備を入手した場合、以下の導線を出す。

* 装備する
* 装備変更へ
* もう一度探索する
* 迷宮へ戻る

表示例。

```text
[装備する]
[装備変更へ]
[もう一度探索する]
```

## 12.4 自動装備はしない

ドロップした装備は、自動では装備しない。

理由は、現在の装備より弱い可能性があるため。

## 13. 公開ログ仕様

## 13.1 公開ログに流す条件

すべてのドロップを公開ログに流すとログが多すぎる。

フェーズ1では以下のみ公開ログに流す。

```text
rarity = rare 以上
```

normal は公開ログに流さない。

## 13.2 公開ログ文言

rare。

```text
【獲得】テストプレイさんが「古びた短剣」を手に入れました！
```

epic。

```text
【獲得】テストプレイさんがエピック装備「親分の腕輪」を手に入れました！
```

legend。

```text
【伝説】テストプレイさんが伝説装備「洞窟王の護符」を手に入れました！
```

フェーズ1では legend は入れなくてよい。

## 13.3 public_logs type

ドロップログの type は以下。

```text
drop
```

importance の目安。

| rarity | importance |
| ------ | ---------: |
| normal |      作成しない |
| rare   |          2 |
| epic   |          3 |
| legend |          3 |

## 16.4 ドロップ率方針

通常敵の rare は 1%〜1.5% 程度。
normal は 3% 程度。
ボスは 15%〜20% 程度。

フェーズ1ではボス再戦不可の場合、ボスドロップを取り逃がす可能性がある。
それが嫌な場合は、ボス撃破時は確定ドロップでもよい。

推奨は以下。

```text
フェーズ1では、ボス初回撃破時はドロップ確定
```

理由は、ボス再戦不可にするなら、低確率ドロップは不満になりやすいため。

採用する場合、ボスの drop_rate は 100.0 にする。

ただし、レア装備の希少性を残すなら 20.0 でもよい。

## 17. ボスドロップ方針

## 17.1 案A ボスドロップ確率制

ボス撃破時にも通常通り drop_rate で判定する。

メリット。

* レア感がある
* 周回要素と相性がよい

デメリット。

* フェーズ1ではボス再戦不可のため、取り逃がすと不満になる

## 17.2 案B ボス初回撃破時は確定

ボス撃破時は専用装備を確定で付与する。

メリット。

* ボス撃破報酬として分かりやすい
* エリア突破の達成感が出る
* 再戦不可でも不満が少ない

デメリット。

* 全員が持つため希少性は下がる

## 17.3 推奨

フェーズ1では案Bを推奨する。

つまり、ボスドロップは確定でよい。

```text
草原の大スライム撃破 → 大スライムの護符
小鬼の森の親分撃破 → 親分の腕輪
洞窟の番人撃破 → 番人の護符
```

将来的にボス再戦やレア周回を入れる場合、確率制に変更してもよい。

## 24. テスト項目

## 24.1 ドロップ抽選テスト

* 戦闘勝利時のみドロップ抽選される
* 敗北時はドロップしない

* ドロップ時に character_items が作成される
* acquired_from が drop になる
* is_equipped は false になる
* equipped_slot は null になる
* 同じドロップ装備を複数所持できる

## 24.3 戦闘結果表示テスト

* ドロップありの場合、装備名が表示される
* rarity が表示される
* 補正値が表示される
* 装備変更への導線が表示される
* ドロップなしの場合、表示が崩れない

## 24.4 公開ログテスト

* normal ドロップでは公開ログが出ない
* rare ドロップでは公開ログが出る
* epic ドロップでは公開ログが出る
* public_logs.type が drop になる

## 24.5 ボスドロップテスト

## 26. バランス調整方針

## 26.1 通常敵ドロップ

通常敵のドロップ率は低めにする。

目安。

| rarity |  通常敵ドロップ率 |
| ------ | --------: |
| normal |     3%〜5% |
| rare   | 0.8%〜1.5% |
| epic   |      原則なし |
| legend |      原則なし |

## 26.2 ボスドロップ

フェーズ1では、ボス再戦不可なら確定ドロップを推奨する。

| ボス       | ドロップ     |
| -------- | -------- |
| 草原の大スライム | 大スライムの護符 |
| 小鬼の森の親分  | 親分の腕輪    |
| 洞窟の番人    | 番人の護符    |

## 26.3 ドロップ装備の強さ

ドロップ装備は、同レベル帯の店売り装備より少し強い程度にする。

例。

店売り「俊足の指輪」

```text
AGI +5
```

ドロップ「うさぎの足飾り」

```text
AGI +4 / LUK +3
```

単純な上位互換にしすぎず、少し個性を出す。

## 28. 結論

今回の実装では、戦闘に「装備が落ちる楽しさ」を追加する。

これにより、ヴァルゼリアの戦闘ループは以下のように強化される。

```text
戦う
↓
EXPとGOLDを得る
↓
たまに装備が落ちる
↓
装備変更する
↓
さらに探索したくなる
```

GOLDで買う装備は安定成長。
敵から落ちる装備は探索の楽しみ。

この2つを分けることで、武具屋と迷宮探索の両方に意味が生まれる。

フェーズ1では、まずは敵から直接装備が落ちるシンプルな仕様でよい。
素材、鑑定、強化、取引は後回しにする。
