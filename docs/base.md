# FFA風ブラウザRPG 初期実装仕様書

## 1. 概要

### 1.1 目的

昔流行したCGIゲーム系FFAの遊び心地を、現代のWebアプリとして再構築する。

本作は、プレイヤーが冒険者キャラクターを作成し、戦闘、レベルアップ、装備購入、転職、ランキング上昇を繰り返すブラウザRPGである。

初期実装では、複雑なリアルタイム戦闘や大規模コミュニティ機能は入れず、FFAの核である以下のループを成立させることを最優先とする。

```text
ログイン
↓
キャラクター作成
↓
戦闘
↓
経験値・お金獲得
↓
レベルアップ
↓
装備購入・強化
↓
転職
↓
より強いエリアへ挑戦
↓
ランキング上位を目指す
```

### 1.2 初期実装のコンセプト

初期実装のコンセプトは、次の3点とする。

1. 戦うたびに少しずつ強くなる
2. 次に目指す目標が常に見える
3. 他プレイヤーの存在をランキングで感じられる

### 1.3 対象プラットフォーム

PCブラウザ
スマートフォンブラウザ

ネイティブアプリ化は初期実装では対象外とする。

### 1.4 想定技術構成

既存開発方針に合わせる場合、以下を想定する。

バックエンド：Laravel
フロントエンド：Blade または Vue / React
DB：MySQL または PostgreSQL
認証：Google OAuth
画像保存：ローカルストレージまたはS3互換ストレージ
デプロイ：VPS または PaaS

ただし、初期実装ではフロントエンドを過度に複雑化せず、Laravel Blade中心でも問題ない。

## 3. 認証仕様

## 3.1 ログイン方式

ログインはGoogleアカウント連携を使用する。

メールアドレスとパスワードによる独自認証は、初期実装では行わない。

### 3.2 認証フロー

1. ユーザーがログイン画面にアクセスする
2. Googleでログインボタンを押す
3. Google OAuth認証画面に遷移する
4. 認証成功後、アプリ側に戻る
5. Googleアカウント情報をもとにユーザーデータを作成または取得する
6. キャラクター未作成の場合、キャラクター作成画面へ遷移する
7. キャラクター作成済みの場合、ホーム画面へ遷移する

### 3.3 取得するGoogleアカウント情報

初期実装で取得する情報は以下とする。

* GoogleアカウントID
* メールアドレス
* 表示名
* プロフィール画像URL

### 3.4 ユーザー管理方針

Googleアカウント1つにつき、ゲームアカウント1つを紐づける。

初期実装では、1ユーザーにつき作成できるキャラクターは1体のみとする。

### 3.5 ログアウト

ログアウト時はアプリ側のセッションを破棄する。

Googleアカウント自体からのログアウトは行わない。

## 4. ユーザー・キャラクター仕様

## 4.1 ユーザーとキャラクターの関係

ユーザーは認証情報を管理する単位。
キャラクターはゲーム内で育成する冒険者データを管理する単位。

```text
users
  └ characters
```

初期実装では、1ユーザーに対して1キャラクターのみ作成可能とする。

### 4.2 キャラクター作成項目

初回ログイン後、キャラクター未作成のユーザーはキャラクター作成画面へ遷移する。

入力項目は以下とする。

| 項目      | 内容            | 備考             |
| ------- | ------------- | -------------- |
| キャラクター名 | ゲーム内表示名       | 重複可否は後述        |
| 性別      | 男性、女性、その他、未設定 | ゲームバランスには影響しない |
| アイコン    | 初期アイコンから選択    | 後から変更可能        |
| 初期職業    | 戦士、魔法使い、僧侶、盗賊 | 初期4職から選択       |

### 4.3 キャラクター名ルール

キャラクター名は以下のルールとする。

* 2文字以上16文字以内
* 公序良俗に反する名前は禁止
* 管理者が不適切と判断した名前は変更対象
* 初期実装では重複を許可する
* ランキング上ではキャラクターIDを内部的に持つため、同名でも区別可能とする

### 4.4 初期ステータス

初期ステータスは職業によって補正を持たせる。

共通初期値は以下とする。

| 項目  |  初期値 |

| レベル |    1 |
| 経験値 |    0 |
| 所持金 | 100G |
| HP  |  100 |
| 攻撃  |   10 |
| 防御  |    8 |
| 素早さ |    8 |
| 魔力  |    8 |
| 運   |    5 |

職業補正は以下とする。

| 職業   |  HP | 攻撃 | 防御 | 素早さ | 魔力 |  運 |

| 魔法使い | -10 | -2 | -2 |   0 | +6 |  0 |

| 盗賊   |   0 | +1 | -1 |  +5 |  0 | +2 |

## 5. 職業仕様

## 5.1 初期職業

初期職業は以下とする。

| 職業ID | 職業名  | 特徴                  |
| ---: | ---- | ------------------- |
|    1 | 戦士   | HP、攻撃、防御が高い         |
|    2 | 魔法使い | 魔力が高く、魔法攻撃が得意       |
|    3 | 僧侶   | 回復系スキルを持つ安定型        |
|    4 | 盗賊   | 素早さと運が高く、回避や追加報酬に強い |

### 5.2 上位職業

初期実装で入れる上位職は以下とする。

| 職業ID | 職業名  | 解放条件                    |
| ---: | ---- | ----------------------- |
|  101 | ナイト  | 戦士ジョブレベル10              |
|  102 | 賢者   | 魔法使いジョブレベル10、僧侶ジョブレベル10 |
|  103 | アサシン | 盗賊ジョブレベル10              |
|  104 | 魔法剣士 | 戦士ジョブレベル5、魔法使いジョブレベル5   |

### 5.3 ジョブレベル

キャラクターレベルとは別に、現在職業ごとのジョブ経験値とジョブレベルを持つ。

戦闘に勝利すると、キャラクター経験値とは別にジョブ経験値を獲得する。

ジョブレベルが上がると以下の効果を得る。

* 職業スキルの発動率上昇
* 転職条件の解放
* ステータス補正の増加

### 5.4 転職仕様

転職は専用画面から行う。

転職時の仕様は以下とする。

* キャラクターレベルは維持する
* 所持金は維持する
* 装備は維持する
* 職業のみ変更される
* 現在職業のジョブ経験値は保存される
* 転職後は新しい職業のジョブレベルが反映される
* 転職によって装備条件を満たさなくなった場合、その装備は自動的に外れる

### 5.5 職業スキル

初期実装では、戦闘中に自動で発動するパッシブ型スキルとする。

手動選択式のスキルは初期実装では入れない。

| 職業   | スキル名 | 効果                 |
| ---- | ---- | ------------------ |
| 戦士   | 会心斬り | 一定確率で通常攻撃の1.8倍ダメージ |
| 魔法使い | ファイア | 魔力依存の追加ダメージ        |
| 僧侶   | ヒール  | 戦闘中に一度だけHPを回復する可能性 |
| 盗賊   | 急所突き | 一定確率で防御を一部無視して攻撃   |
| ナイト  | 鉄壁   | 一定確率で被ダメージを半減      |
| 賢者   | 聖なる光 | 魔法攻撃と回復の両方を行う      |
| アサシン | 暗殺   | 低確率で大ダメージ          |
| 魔法剣士 | 魔法剣  | 攻撃と魔力を合算した追加ダメージ   |

## 6. 戦闘仕様

## 6.1 戦闘方式

戦闘は自動ターン制とする。

プレイヤーは戦闘開始前にエリアを選択し、戦闘開始ボタンを押す。
戦闘開始後は、内部処理で勝敗を決定し、結果をテキストログとして表示する。

リアルタイム操作は行わない。

### 6.2 戦闘フロー

```text
エリア選択
↓
敵抽選
↓
先攻判定
↓
ターン開始
↓
攻撃処理
↓
スキル発動判定
↓
HP減少
↓
勝敗判定
↓
決着まで繰り返し
↓
結果表示
↓
経験値・お金・ジョブ経験値・ドロップ判定
```

### 6.3 先攻判定

先攻は素早さをもとに判定する。

基本仕様は以下とする。

```text
プレイヤー素早さ + ランダム値
敵素早さ + ランダム値
```

数値が高い方が先攻となる。

### 6.4 ダメージ計算

初期実装では、複雑な計算式にしすぎない。

物理攻撃の基本ダメージは以下とする。

```text
基本ダメージ = 攻撃側の攻撃力 - 防御側の防御力 × 0.5
```

最低ダメージは1とする。

乱数補正として、最終ダメージに80％から120％のブレを持たせる。

```text
最終ダメージ = 基本ダメージ × 乱数補正
```

魔法攻撃は以下とする。

```text
魔法ダメージ = 攻撃側の魔力 × スキル倍率 - 防御側の防御力 × 0.2
```

### 6.5 回避判定

素早さの差によって低確率で回避が発生する。

初期実装では、回避率の上限を20％とする。

```text
回避率 = 5％ + (防御側素早さ - 攻撃側素早さ) × 0.5％
```

最低回避率は3％、最大回避率は20％とする。

### 6.6 会心判定

運によって会心率が上がる。

```text
会心率 = 5％ + 運 × 0.2％
```

最大会心率は25％とする。

会心時のダメージ倍率は1.5倍とする。

### 6.7 勝利時の報酬

戦闘に勝利した場合、以下を得る。

* 経験値
* ジョブ経験値
* お金
* ドロップアイテム抽選
* 勝利数 +1

### 6.8 敗北時の処理

敗北時は以下の処理を行う。

* 敗北数 +1
* 経験値は獲得しない
* ジョブ経験値は獲得しない
* お金は失わない
* 装備は失わない
* HPは戦闘後に全回復する

初期実装では、敗北ペナルティを軽くする。
理由は、序盤離脱を防ぐためである。

### 6.9 戦闘ログ

戦闘結果はテキストログで表示する。

表示例は以下。

```text
はじまりの草原に向かった。

スライムが現れた！

あなたの攻撃！
スライムに18ダメージ！

スライムの攻撃！
あなたに7ダメージ！

戦士のスキル「会心斬り」が発動！
スライムに34ダメージ！

スライムを倒した！

経験値 12 を獲得。
ジョブ経験値 3 を獲得。
18G を手に入れた。
```

### 6.10 戦闘制限

初期実装では、スタミナ制は導入しない。

ただし、連打対策として以下を導入する。

* 戦闘開始後、次の戦闘まで3秒のクールタイム
* 同時戦闘リクエストの二重送信防止
* サーバー側で最終戦闘時刻を確認

## 8. レベルアップ仕様

## 8.1 キャラクターレベル

戦闘勝利で経験値を獲得し、一定値に達するとレベルアップする。

必要経験値は以下の式を基本とする。

```text
次レベル必要経験値 = 現在レベル ^ 2 × 20 + 現在レベル × 30
```

例。

| 現在レベル | 次レベル必要経験値 |
| ----: | --------: |
|     1 |        50 |
|     2 |       140 |
|     3 |       270 |
|     4 |       440 |
|     5 |       650 |

### 8.2 レベルアップ時の成長

レベルアップ時、基礎ステータスが上昇する。

成長量は職業によって変動する。

例。

| 職業   |     HP |    攻撃 |    防御 |   素早さ |    魔力 |     運 |
| ---- | -----: | ----: | ----: | ----: | ----: | ----: |
| 戦士   | +8から12 | +2から4 | +2から3 | +0から2 | +0から1 | +0から1 |
| 魔法使い |  +4から8 | +0から2 | +0から1 | +0から2 | +3から5 | +0から1 |
| 僧侶   | +6から10 | +0から2 | +1から3 | +0から2 | +2から4 | +0から2 |
| 盗賊   |  +5から9 | +1から3 | +0から2 | +2から4 | +0から2 | +1から3 |

### 8.3 レベル上限

初期実装のレベル上限は50とする。

レベル50到達後も戦闘は可能だが、経験値は増えない。

将来的に転生機能を追加する場合、レベル50到達を転生条件にできる。

| epic   | 高性能装備  |
| legend | 最上位装備  |

初期実装では、legend装備は数を絞る。
強すぎる装備を序盤から出すと、成長バランスが崩れるためである。

## 10. ショップ仕様

## 10.1 ショップ種別

初期実装では以下のショップを用意する。

* 武器屋
* 防具屋
* アクセサリー屋

道具屋は初期実装では入れない。
理由は、戦闘後にHP全回復する仕様のため、回復アイテムの優先度が低いためである。

### 10.2 購入

ショップでは、所持金を消費して装備を購入できる。

購入条件は以下。

* 所持金が価格以上ある
* 必要レベルを満たしている
* 必要職業を満たしている
* すでに所持している同一装備でも購入可能

### 10.3 売却

所持装備は売却できる。

売却価格は装備マスターの売却価格を使用する。

装備中のアイテムは売却できない。

### 10.4 装備変更

所持している装備から、武器、防具、アクセサリーを変更できる。

装備変更時、キャラクターの表示ステータスは装備補正込みで再計算する。

## 12. ランキング仕様

* 攻撃力ランキング
* 防御力ランキング

### 12.2 総合ランキング計算

総合ランキングはスコア制とする。

```text
総合スコア =
レベル × 1000
+ 勝利数 × 10
+ 所持金 × 0.1
+ 攻撃力 × 20
+ 防御力 × 20
+ 素早さ × 10
+ 魔力 × 10
+ 運 × 10
```

小数点以下は切り捨てる。

### 12.3 ランキング更新タイミング

ランキングはリアルタイム集計でもよいが、負荷対策として以下を推奨する。

* 戦闘終了時にキャラクターの総合スコアを更新
* ランキング画面では総合スコア順で表示
* キャッシュは初期実装では必須ではない

### 17.2 BattleServiceの責務

BattleServiceに以下を集約する。

* エリア挑戦可否チェック
* 敵抽選
* 戦闘ターン処理
* ダメージ計算
* スキル発動判定
* 勝敗判定
* 報酬計算

### 17.3 ステータス計算

キャラクターの最終ステータスは以下で算出する。

```text
最終ステータス =
基礎ステータス
+ 現在職業補正
+ 装備補正
```

DBに保存するのは基礎ステータスを基本とする。
装備込みの最終ステータスは都度計算する。

理由は、装備変更や転職時にステータスの不整合が起きにくいためである。

## 19. 不正対策・安全設計

## 19.1 戦闘の二重実行防止

戦闘リクエストが連続送信された場合に備え、サーバー側で以下をチェックする。

* 最終戦闘時刻から3秒未満なら拒否
* 同一ユーザーの同時戦闘処理をロック
* フロント側のボタン連打防止だけに依存しない

### 19.2 パラメータ改ざん防止

戦闘時に、フロントから以下の値を受け取らない。

* 攻撃力
* 防御力
* HP
* 獲得経験値
* 獲得金
* 敵ステータス
* ドロップ結果

フロントから受け取るのは、基本的にエリアIDのみとする。
戦闘計算はすべてサーバー側で行う。

### 19.3 Google認証の扱い

Googleアカウントのメールアドレスはユーザー識別に使うが、主キーとしては使わない。

アプリ内では users.id を内部IDとして使用する。

### 19.4 表示名・キャラクター名対策

キャラクター名はHTMLエスケープして表示する。

管理者側で不適切な名前を変更できるようにする。

## 23. 初期実装で特に重要な判断

## 23.1 最初から複雑にしすぎない

FFAは機能を増やしやすいゲームである。
しかし、最初からギルド、PvP、転生、クラフト、イベントを入れると、完成前に設計が膨らみすぎる。

初期実装では、以下に集中する。

```text
戦う
強くなる
装備を買う
転職する
ランキングを見る
```

この5つでゲームとして成立させる。

### 23.2 戦闘はサーバー側で完結させる

戦闘結果をフロント側で計算すると、不正や改ざんの原因になる。

戦闘処理は必ずサーバー側で行う。

### 23.3 ステータスは基礎値と補正値を分ける

キャラクターの基礎値、職業補正、装備補正を分けて扱う。

この設計にしておくと、転職や装備変更による不整合を防ぎやすい。

### 23.4 管理画面は後回しにしすぎない

バランス調整が必要なゲームなので、管理画面はかなり重要である。

特に以下は管理画面から編集可能にする。

* 敵の強さ
* 経験値
* お金
* 装備価格
* 装備補正
* ドロップ率
* 転職条件

### 23.5 ランキングは初期から入れる

ランキングはFFAらしさの中核である。

直接対人戦がなくても、ランキングがあるだけで他プレイヤーの存在を感じられる。

## 26. 結論

初期実装で最も大事なのは、派手な機能を増やすことではない。

重要なのは、以下の育成ループを気持ちよく成立させることである。

```text
戦う
↓
強くなる
↓
装備を買う
↓
転職する
↓
より強い敵に挑む
↓
ランキングが上がる
```

このループが成立すれば、昔のFFAらしい中毒性は再現できる。

初期実装では、戦闘、成長、装備、職業、ランキングの5本柱を優先する。
ギルド、PvP、転生、イベントなどは、プレイヤーが定着してから追加する。
