# ヴァルゼリアの冒険者 — CLAUDE.md

## プロジェクト概要

昔流行したCGIゲーム系FFAの遊び心地を、現代のWebアプリとして再構築するブラウザRPG。
Laravel + Livewire 製。デプロイ先は Xserver（`valzeria.com`）。

**コアループ（最優先で守ること）**
```
ログイン → キャラ作成 → 戦闘 → 経験値・Gold獲得 → レベルアップ
→ 装備購入 → 転職 → より強いエリアへ → ランキング上位を目指す
```

---

## 技術スタック

- **Backend**: Laravel 11 + PHP
- **Frontend**: Livewire v3 + Blade + Alpine.js + Tailwind CSS
- **DB**: MySQL（本番 Xserver）
- **認証**: Google OAuth（1アカウント1キャラ）
- **デプロイ**: `php local_deploy.php`（ZIP → サーバーPOST → 自動展開）

## ゲームの全体設計

### 都市・エリア構成
- **都市数**: 10都市（アークレア → マリネス → エルフィア → グランベルグ → フロストリア → サンドラ → ルミナス → ネクロム → セレスティア → ヴァルゼリア城）
- **エリア数**: 都市ごとに複数のダンジョン、計40以上（Area ID 1〜70が通常ダンジョン、71〜74が特殊、75〜83が街道）
- **解放条件**: 都市ボスを撃破 + ジョブ条件を満たすと次の都市が解放される

### レベル・ステータス設計
- **レベル上限**: 255（2026-07-02裁定。Lv255到達時は超過経験値を0に戻す）
- **ステータス表記（プレイヤー向け）**: HP / SP / ATK(攻撃) / DEF(防御) / MAG(魔力) / SPR(精神) / SPD(素早さ) / LUK(運)
  - 内部カラムには `mp_base` / `str` / `agi` / `luck` 等の旧名が残るが、表示・新規ドキュメントに旧名（MP/STR/AGI）を使わない。DBカラムのリネームは行わない（別途の高リスクタスク扱い）
  - `gold` ではなく `money` カラムを使用（Character モデル参照）
- **最終ステータス = 基礎値 + 職業補正 + 装備補正 + ジョブマスターボーナス**
  - DBには基礎値のみ保存し、補正は都度計算（`CharacterStatusService`）

### 職業システム
- **基本職**: 戦士・魔法使い・僧侶・盗賊
- **上位職**: 条件を満たすと転職可能。転職条件の正仕様は「Lv30以上 + 要求されている職のマスター」（2026-07-02裁定。valzeria_specの「Lv100」は未採用案でありコードへ反映しない）
- **伝説職**: 高難度条件で解放される最上位職
- **ジョブレベル**: 職業ごとに独立した経験値・レベル（マスターで特殊ボーナス付与）。プレイヤー向け表示は「職業レベル」ではなく「職業ランク」（★1〜★10、★10でマスター）
- **転職時**: キャラレベルはLv1に戻り、基礎ステータスは半分になる。所持金・装備・過去のジョブ経験値は維持される

### バランス設計の基準
- **装備の強さ**: 価格帯から強さ階層を逆算して設計（レアリティから直算しない）
- **敵ステータス**: 最終エリアのボスを基準に逆算してカーブを設計
- **装備 vs 敵HP**: 装備の攻撃上限をインフレさせた場合、敵HPも合わせてスケールさせること
- **ショップ価格の目安**: 都市の推奨レベル帯での獲得Goldと照らし合わせて設定
- **合成品**: 店売りの調整と同時に合成専用アイテムも再計算すること（見落とし注意）

### 戦闘設計
- 自動ターン制（サーバー側で完結、フロントからはエリアIDのみ受け取る）
- PRGパターン必須：状態更新後は必ず `redirect()` でGETへ遷移（二重送信防止）
- 二重実行防止: `last_battle_at` で3秒クールタイムチェック

### ドロップ設計
- **通常ドロップ**: 敵ごとにドロップテーブル（`enemy_drops`）で管理
- **素材ドロップ**: 合成素材は別テーブルで管理
- **ドロップ率目安**: normal 5〜10% / rare 1〜3% / epic 0.3〜1% / legend 0.05〜0.2%

## コーディング規約・設計方針

### モデルと命名
- `Character::hasMany` 関係 → `Auth::user()->characters()->first()` で取得（`->character` は使えない）
- Eloquentモデル（DBカラム）はスネークケース、自作DTO・BattleActorなどはキャメルケース
- マイグレーションでカラム追加したら `$fillable`（または `$casts`）も同時に更新

### 同名エンティティへの注意
- 同名の敵・素材が別ダンジョンに存在する場合、`name` だけで検索するとバグになる
- 必ず `dungeon_id`（`area_id`）との複合条件で一意に特定する

### UI・デザイン規則
- **施設画面**: `<x-layouts.facility>` で全画面表示に統一
- **施設ヘッダー**: 背景画像 + 暗いオーバーレイ(`bg-slate-900/60`) + 白テキスト
- **「街へ戻る」ボタン**: レイアウト共通部品に置く（各ビューに個別配置しない）
- **Tailwind**: `min-h-[...]` や `flex-wrap` を使うとカードの縦幅が不均一になるので注意

### TSV・マスターデータ取り込み
- 手動作成TSVは行ごとにタブ数がズレることがある → 取り込み後は代表レコードを必ず確認
- SeederでTSVフォーマットを変更する場合はヘッダー行から動的にマッピング（`array_combine`）
- Seederで `truncate()` を使う場合は外部キーチェックを無効化してから実行

## デプロイ手順

```powershell
php local_deploy.php
```

- `npm run build` → ZIP圧縮 → `valzeria.com/server_deploy_api.php` へPOST送信
- サーバー側で展開 → マイグレーション自動実行 → Seeder自動実行
- **除外**: `.git`, `vendor`, `node_modules`, `storage`, `.env`, `backups/`

### ZIPデプロイの罠
- マイグレーションファイルをリネームした場合、旧ファイルがサーバーに残り二重実行エラーになる
- 対策: 旧ファイル名のままダミー（空の `up()`/`down()`）を作成して上書きデプロイ

## 作業時の注意事項

1. **バランス調整はスケール全体で考える** — 装備・敵・Goldのどれか1つだけ変えると他が崩れる
2. **1アカウント1キャラ** — `CharacterCreate` と `CharacterSelect` で二重ガード済み
3. **`last_seen_at` でオンライン判定** — 5分以内にアクセスしたキャラのみ「街にいる冒険者」に表示
4. **コミット前にデプロイ不要な一時スクリプト(`scratch_*.php` 等)は除外するか確認する**
5. **新機能追加より核ループの品質を優先** — ギルド・PvP・転生・イベントは後回し
